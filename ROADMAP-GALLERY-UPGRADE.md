# 写真图集网站升级路线图

本文档把后续升级拆成“需求 -> 升级点 -> 具体措施 -> 验收标准”，并按难度由低到高排列。目标是让当前项目从“能浏览本地图集的网站”，逐步升级为“可维护、可搜索、可扩展的个人媒体资产管理系统”。

## L1 基础体验优化

这一层不改变系统架构，主要是在现有 `server.js + gallery.json + app.js` 基础上补功能。适合先做，收益明显，风险较低。

### 1. 搜索入口

**需求**

图库变大后，只靠滚动查找模特和作品效率低。用户需要快速输入关键词找到目标。

**升级点**

增加全站搜索框，支持搜索模特名、作品名、目录名。

**具体措施**

- 在顶部工具栏增加搜索输入框。
- 前端基于当前 `gallery` 数据做即时过滤。
- 首页搜索模特；进入模特页后搜索作品；也可以支持全局搜索结果列表。
- 搜索词保存到当前页面状态，不必永久保存。

**验收标准**

- 输入“剧照”能直接看到对应入口。
- 输入作品标题片段能筛出匹配作品。
- 清空搜索后恢复原列表。
- 移动端搜索框不挤压现有按钮。

### 2. 排序按钮

**需求**

公开图集网站通常支持按最新、最多图片、名称排序。当前排序主要依赖文件夹名称，不够灵活。

**升级点**

增加排序模式。

**具体措施**

- 首页支持：按名称、作品数、图片数、最近修改时间排序。
- 作品页支持：按名称、图片数、视频数、最近修改时间排序。
- 服务端扫描时为 model/work 增加 `mtime`、`totalImageCount`、`totalVideoCount`。
- 前端保存排序偏好到 `localStorage`。

**验收标准**

- 点击排序按钮后列表立即变化。
- 刷新页面后保持上次排序方式。
- 没有图片或视频的目录也能稳定排序。

### 3. 最近观看

**需求**

图集很多时，经常需要回到刚看过的作品。

**升级点**

记录最近打开的模特、作品和图片。

**具体措施**

- 前端在进入作品详情或一级散图目录时写入 `localStorage`。
- 首页增加“最近观看”区域，显示最近 10 个入口。
- 记录字段：标题、路径 hash、封面、访问时间。
- 重复访问同一项时更新到最前。

**验收标准**

- 打开任意作品后，首页能看到最近观看卡片。
- 点击最近观看可以回到对应位置。
- 最多保留 10 或 20 条，不无限增长。

### 4. 收藏功能

**需求**

用户需要标记常看的模特或作品。

**升级点**

增加收藏按钮和收藏列表。

**具体措施**

- 作品详情页和模特页增加收藏按钮。
- 收藏数据先存 `localStorage`，后续可迁移到服务端。
- 首页增加“收藏”视图或工具栏按钮。
- 收藏数据包括：id、标题、封面、类型、路径。

**验收标准**

- 收藏后按钮状态变化。
- 刷新后收藏仍保留。
- 可以取消收藏。
- 收藏列表点击可跳转。

## L2 媒体加载和性能优化

这一层主要解决大图库访问速度、移动端流量和内存压力。改动范围中等，但仍可在现有架构上完成。

### 5. 统一缩略图系统

**需求**

首页卡片、轮播图、作品封面如果直接使用原图，会造成加载慢、带宽高、移动端内存压力大。

**升级点**

生成统一尺寸缩略图，并让列表页优先加载缩略图。

**具体措施**

- 新建 `data/thumbnails` 目录。
- 为图片生成小图，例如：
  - 首页封面：`480px` 宽
  - 轮播图：`960px` 宽
  - 详情预览：可选 `720px` 宽
- 缩略图文件名使用源路径 hash，避免中文和特殊字符问题。
- 服务端扫描时为图片增加 `thumb` 字段。
- 前端列表页使用 `thumb`，灯箱和详情大图仍使用原图。

**验收标准**

- 首页不再直接加载大原图。
- 缩略图缺失时可以自动生成或回退原图。
- 生成后再次刷新不重复生成。
- 移动端打开首页速度明显提升。

### 6. 图片懒加载增强

**需求**

详情页图片很多时，即使使用浏览器原生懒加载，仍可能创建过多 DOM 节点。

**升级点**

改为分批渲染或虚拟列表。

**具体措施**

- 第一阶段：每次只渲染前 30 或 50 张，滚动到底部再追加。
- 第二阶段：使用虚拟列表，只保留视口附近图片 DOM。
- 保留当前“每行数量”设置。
- 灯箱索引仍对应完整图片数组。

**验收标准**

- 500 张图片的图集不会明显卡顿。
- 滚动时图片逐步加载。
- 点击任意可见图片仍能进入正确灯箱索引。

### 7. 视频信息展示

**需求**

公开媒体网站通常展示视频时长、大小、分辨率。当前视频只显示标题。

**升级点**

为视频生成元数据。

**具体措施**

- 使用 `ffprobe` 读取视频时长、宽高、编码、大小。
- 写入 gallery 数据：`duration`、`width`、`height`、`size`。
- 前端在视频卡片或视频标题旁显示：
  - 时长
  - 分辨率
  - 文件大小
- 没有 `ffprobe` 时优雅降级。

**验收标准**

- 视频下方显示类似 `03:21 / 1920x1080 / 128MB`。
- 没有元数据时不报错。
- 刷新目录后信息稳定。

### 8. 视频分段加载预研

**需求**

公网或 ZeroTier 访问大视频时，直接播放大文件可能拖慢或跳转不顺畅。

**升级点**

评估 HLS 分段或保留 HTTP Range 播放。

**具体措施**

- 当前保留 HTTP Range，这是浏览器播放 mp4 的基础。
- 对特别大的视频，可选择生成 HLS：
  - `.m3u8`
  - `.ts` 或 fragmented mp4
- 增加配置开关，不默认全量转码。
- 只对用户手动标记的视频生成分段。

**验收标准**

- 普通 mp4 仍可直接播放。
- 大视频可选择 HLS 播放。
- 不因为后台转码拖慢网站启动。

## L3 数据结构和目录兼容升级

这一层解决“目录层级不统一”的根问题。当前已经为一级散图、二级空目录、三级目录做过补丁，但长期应抽象为统一媒体集合模型。

### 9. 统一 Collection 模型

**需求**

真实文件夹结构不一定永远是“模特 / 作品 / 图片”。有些目录是一级散图，有些是多层目录，有些只有视频。

**升级点**

从固定 model/work 结构升级为通用 collection 树。

**具体措施**

- 服务端扫描任意目录时生成统一结构：
  - `id`
  - `title`
  - `pathParts`
  - `level`
  - `images`
  - `videos`
  - `children`
  - `cover`
  - `imageCount`
  - `videoCount`
  - `descendantCount`
- 前端根据当前节点判断：
  - 有子节点：展示子节点网格
  - 有媒体：展示媒体详情
  - 两者都有：顶部展示子目录，下方展示本级媒体，或提供 Tab
- 保留“模特”作为首页一级分类的显示名称，但内部不再写死。

**验收标准**

- 一级散图目录直接显示图片。
- 二级目录下面还有三级目录时正常继续进入。
- 同一目录同时有图片和子目录时不会丢数据。
- 面包屑可展示任意层级。

### 10. 扫描错误报告

**需求**

大图库中可能存在损坏图片、无法读取目录、权限问题、异常文件名。

**升级点**

增加扫描日志和错误报告。

**具体措施**

- 扫描时记录：
  - 无法访问的目录
  - 无法识别的媒体
  - 生成缩略图失败
  - 视频封面生成失败
- 写入 `data/scan-report.json`。
- 前端增加管理入口查看报告。

**验收标准**

- 刷新目录后生成报告。
- 有问题的文件不会中断整体扫描。
- 管理页能看到错误路径和错误原因。

### 11. 空目录和异常目录治理

**需求**

有些目录没有图片、没有视频，也没有有效子目录，会造成空页面。

**升级点**

自动识别空目录，并在管理页集中展示。

**具体措施**

- 扫描时标记 `isEmpty`。
- 前端空状态显示更具体原因：
  - 没有媒体
  - 只有子目录但子目录也为空
  - 有文件但不是支持格式
- 管理页提供“空目录列表”。

**验收标准**

- 空页面信息准确。
- 可以快速定位真实文件夹。
- 不影响正常目录浏览。

## L4 索引数据库升级

这一层是关键架构升级。目标是从“每次扫描文件夹生成大 JSON”转为“增量索引 + 查询 API”。

### 12. SQLite 增量索引

**需求**

图库越大，全量扫描越慢，启动和刷新成本越来越高。

**升级点**

引入 SQLite 保存索引。

**具体措施**

- 新建 `data/gallery.db`。
- 建表：
  - `collections`
  - `media`
  - `covers`
  - `scan_state`
  - `user_marks`
- 以文件夹路径、mtime、文件数量判断是否需要重扫。
- 只重扫变化的目录。
- 继续导出兼容前端的 JSON，或直接提供查询 API。

**验收标准**

- 第一次扫描建立完整索引。
- 后续只修改一个目录时，刷新速度明显加快。
- 数据库损坏时可以删除重建。

### 13. 查询 API

**需求**

前端一次性加载完整 gallery JSON，数据大了会增加内存和网络负担。

**升级点**

改为按需请求。

**具体措施**

- 新增接口：
  - `GET /api/collections/root`
  - `GET /api/collections/:id`
  - `GET /api/search?q=`
  - `GET /api/recent`
  - `GET /api/favorites`
- 首页只请求一级目录。
- 进入目录时再请求当前目录数据。
- 搜索由服务端查询 SQLite。

**验收标准**

- 首页不需要下载完整图库数据。
- 搜索大图库仍然快速。
- 浏览任意层级时按需加载。

### 14. 后台扫描任务

**需求**

刷新目录可能耗时，前端不应该一直等待。

**升级点**

把扫描改成后台任务。

**具体措施**

- `POST /api/scan` 创建扫描任务。
- `GET /api/scan/status` 返回进度。
- 前端显示进度：
  - 已扫描目录数
  - 已处理文件数
  - 当前目录
  - 错误数量
- 扫描完成后自动刷新页面数据。

**验收标准**

- 点击刷新后页面不卡死。
- 可以看到扫描进度。
- 扫描失败有明确错误。

## L5 用户功能和管理后台

这一层让网站从浏览器变成个人图库管理工具。

### 15. 管理页

**需求**

随着功能变多，需要集中管理索引、缓存、错误和设置。

**升级点**

增加 `/admin` 或 `#/admin` 管理页。

**具体措施**

- 管理页包含：
  - 刷新索引
  - 重建缩略图
  - 清理轮播缓存
  - 查看扫描报告
  - 查看空目录
  - 查看视频封面生成状态
  - 查看系统路径配置
- 管理操作需要二次确认。

**验收标准**

- 管理页可以查看当前 `PHOTOS_DIR`、`DATA_DIR`。
- 可以触发扫描和缩略图重建。
- 危险操作有确认。

### 16. 用户标记系统

**需求**

本地图库是长期使用的，用户需要个人标记。

**升级点**

收藏、隐藏、评分、备注、标签。

**具体措施**

- 数据保存到 SQLite：
  - `favorite`
  - `hidden`
  - `rating`
  - `note`
  - `tags`
- 前端增加：
  - 收藏按钮
  - 隐藏按钮
  - 标签输入
  - 评分控件
- 搜索和筛选支持这些字段。

**验收标准**

- 标记刷新后不丢失。
- 可以筛选收藏、隐藏、评分高的作品。
- 隐藏项默认不显示，但可在管理页恢复。

### 17. 观看进度

**需求**

图片多的作品可能一次看不完，需要恢复到上次位置。

**升级点**

记录作品看到第几张、视频播放到哪里。

**具体措施**

- 图片灯箱切换时记录当前 index。
- 作品详情页顶部显示“继续看到第 N 张”。
- 视频监听 `timeupdate`，定期记录播放位置。
- 点击继续播放恢复到上次时间。

**验收标准**

- 离开作品再回来，可以继续上次图片位置。
- 视频能恢复播放进度。
- 用户可以清除观看进度。

## L6 高级展示和推荐

这一层偏体验增强，适合在底层索引和缩略图稳定后做。

### 18. 智能轮播和亮点推荐

**需求**

当前轮播随机选图，后续可以更智能地展示高质量内容。

**升级点**

建立亮点图池。

**具体措施**

- 优先选择：
  - 收藏作品
  - 最近新增
  - 高评分
  - 同尺寸图片
  - 清晰度高的图片
- 允许用户手动固定某些图进入轮播。
- 管理页提供“重新生成轮播”按钮。

**验收标准**

- 轮播图尺寸协调。
- 每小时自动刷新仍保留。
- 手动固定的图不会被随机替换。

### 19. 相似作品推荐

**需求**

公开图集网站常有“你可能喜欢”或“同模特其他作品”。

**升级点**

详情页底部推荐相关作品。

**具体措施**

- 简单版本：
  - 同模特其他作品
  - 同标签作品
  - 最近新增作品
- 高级版本：
  - 根据收藏和观看历史推荐。

**验收标准**

- 作品详情底部显示 6 到 12 个推荐。
- 点击推荐能进入作品。
- 不推荐当前作品自身。

### 20. 时间线视图

**需求**

用户可能希望按新增时间或拍摄时间浏览。

**升级点**

增加时间线页面。

**具体措施**

- 根据文件夹 mtime 或目录名中的日期解析时间。
- 按年份、月份分组。
- 支持“最近新增”入口。

**验收标准**

- 可以看到最近新增的作品。
- 日期解析失败时回退 mtime。
- 时间线不影响原目录浏览。

## 推荐实施顺序

建议按下面顺序推进，避免一开始就做大重构：

1. 搜索入口
2. 排序按钮
3. 最近观看
4. 收藏功能
5. 统一缩略图系统
6. 图片分批渲染
7. 扫描错误报告
8. 统一 Collection 模型
9. SQLite 增量索引
10. 查询 API
11. 后台扫描任务
12. 管理页
13. 用户标记系统
14. 观看进度
15. 智能推荐和时间线

## 架构原则

- 不让前端一次性承担全部图库数据。
- 不让启动流程依赖昂贵的全量扫描。
- 原图只在需要看细节时加载。
- 缩略图、视频封面、轮播图都应缓存并可重建。
- 文件夹结构不可信，程序要兼容多层级和散图目录。
- 用户数据和图库索引分开存储，避免重建索引时丢收藏、备注和观看进度。
