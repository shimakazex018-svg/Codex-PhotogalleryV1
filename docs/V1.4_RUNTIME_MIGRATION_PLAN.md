# V1.4 Runtime Migration Plan

## 1. Scope and constraints

本文件是 V1.4 第一步的只读环境审计与迁移计划。审计期间：

- 未启动新旧网站；
- 未修改代码或启动脚本；
- 未打开或修改生产 SQLite；
- 未复制、移动、删除或递归扫描真实媒体；
- 未修改端口、`PHOTOS_DIR`、`DATA_DIR` 或其他运行变量。

为避免将真实机器路径写入 Git，本文使用以下占位符：

| 占位符 | 含义 |
|---|---|
| `<legacy-project-root>` | 旧项目实际运行目录 |
| `<legacy-data-root>` | 旧项目的生产 data 目录 |
| `<project-root>` | Codex-PhotogalleryV1 项目目录 |
| `<media-root>` | 外部真实图片/视频目录 |
| `<v1-runtime-root>` | 推荐的新项目专用运行数据目录，位于 Git 跟踪范围外 |
| `<ffmpeg-bin>` | FFmpeg 和 FFprobe 所在目录 |

## 2. Current legacy environment structure

```text
<legacy-project-root>/
├─ server.js
├─ gallery-db.js
├─ duplicates-worker.js
├─ start-server-48101.cmd
└─ data/                         # <legacy-data-root>
   ├─ gallery.db
   ├─ gallery.json
   ├─ thumbnails/
   ├─ video-thumbnails/
   ├─ highlight-carousel/
   ├─ highlight-carousel.json
   ├─ video-metadata.json
   ├─ logs/
   └─ hls/

<media-root>/                    # 外部媒体盘，不在项目或 Git 中
```

旧主要启动脚本的审计事实：

- `PORT=48101`
- `HOST=0.0.0.0`
- `PHOTOS_DIR` 指向外部 `<media-root>`
- `DATA_DIR=<legacy-data-root>`
- `FFMPEG_PATH` 指向 `<ffmpeg-bin>/ffmpeg.exe`
- `ALLOW_REMOTE_DELETE=1`
- 旧简化启动脚本优先使用用户目录中的固定 Node 可执行文件

旧环境数据盘点（2026-07-11 审计快照）：

| 内容 | 数量/大小 | 当前用途 | 迁移分类 |
|---|---:|---|---|
| `gallery.db` | 1,169,600,512 bytes（约 1.09 GiB） | 当前主索引、收藏/最近、扫描状态、媒体哈希 | 必须迁移 |
| `thumbnails/` | 9,245 文件，约 607 MiB | 图片 480/720/960 缩略图 | 条件迁移，可重建 |
| `gallery.json` | 373,870,156 bytes（约 357 MiB） | 旧 JSON 产物；当前恢复逻辑已禁用 | 默认不迁移，单独归档可选 |
| `video-thumbnails/` | 549 文件，约 39 MiB | 视频 poster 文件 | 条件迁移 |
| `highlight-carousel/` | 20 文件，约 1.8 MiB | 当前小时轮播缓存 | 不必迁移，可重建 |
| `logs/` | 10 文件，约 180 KiB | 历史运行/访问日志 | 不迁移到新 runtime；可单独归档 |
| `video-metadata.json` | 约 157 KiB | FFprobe 元数据缓存 | 建议迁移，减少重新探测 |
| `highlight-carousel.json` | 约 9 KiB | 轮播缓存描述 | 不必迁移，可重建 |
| `hls/` | 空 | 可选 HLS 输出 | 当前无需迁移 |

旧 data 总量约 2.07 GiB。审计时数据库旁未发现 `gallery.db-wal` 或 `gallery.db-shm`。审计时端口 `48101` 没有监听进程，但实施前必须重新确认，不能依赖该快照。

外部媒体根目录存在，审计到约 40 个一级目录。为避免对大媒体盘造成额外 I/O，本次没有递归统计文件数或总容量。

## 3. Current V1 runtime expectations

V1 代码按以下方式解析环境：

| 配置 | 默认或派生位置 | 说明 |
|---|---|---|
| `PORT` | `48101` | HTTP 端口 |
| `HOST` | `0.0.0.0` | 监听地址 |
| `PHOTOS_DIR` | `<project-root>/photos` | 原始媒体根目录 |
| `DATA_DIR` | `<project-root>/data` | SQLite、日志和生成缓存根目录 |
| `gallery.db` | `DATA_DIR/gallery.db` | 不能单独配置文件名 |
| 图片缩略图 | `DATA_DIR/thumbnails` | 固定派生路径 |
| `THUMBNAILS_DIR` | `DATA_DIR/video-thumbnails` | 实际是视频 poster 目录，可覆盖 |
| `HLS_DIR` | `DATA_DIR/hls` | HLS 目录，可覆盖 |
| 轮播文件 | `DATA_DIR/highlight-carousel` | 固定派生路径 |
| 轮播描述 | `DATA_DIR/highlight-carousel.json` | 固定派生路径 |
| 视频元数据 | `DATA_DIR/video-metadata.json` | 固定派生路径 |
| 日志 | `DATA_DIR/logs` | 固定派生路径 |
| `TRASH_DIR` | `PHOTOS_DIR` 同级的回收目录 | 删除重复媒体的目标 |
| `FFMPEG_PATH` | `ffmpeg` | 可执行文件名或绝对路径 |
| `FFPROBE_PATH` | 从 FFmpeg 同目录推导或 `ffprobe` | 可显式覆盖 |
| `DUPLICATE_BATCH_SIZE` | `100`，范围限制 1–1000 | 查重批次大小 |
| `ALLOW_REMOTE_DELETE` | `0` | 新项目安全默认值 |

`.env.example` 只提供格式，应用不会自动读取 `.env`。

重要启动脚本差异：

- `start-server-48101.cmd` 会固定设置 `PORT`、`HOST` 和项目内 `DATA_DIR`。
- 该脚本只在 `PHOTOS_DIR` 未预设时使用项目内 `photos`。
- `start-site.cmd`、`start-site.ps1` 不覆盖 `PHOTOS_DIR` 或 `DATA_DIR`，适合在外部先注入环境变量。
- V1 脚本不再绑定旧用户目录中的 Node 可执行文件，要求 `node` 可从 PATH 解析。

## 4. Node and FFmpeg dependencies

### Node.js

- 旧环境已验证版本：Node `v24.14.0`。
- 数据层直接 `require("node:sqlite")` 并使用 `DatabaseSync`。
- 当前 Codex 命令环境中的普通 `node` 不可用，但旧 bundled Node `v24.14.0` 可执行。
- V1 当前启动脚本只调用 `node`，所以正式恢复前必须安装/配置可从 PATH 解析的兼容 Node 24.x，或使用项目外托管器显式指定 Node 可执行文件。

实施时的最低预检：

```powershell
node -v
node -e "const { DatabaseSync } = require('node:sqlite'); console.log(Boolean(DatabaseSync))"
```

首次切换建议保持 Node `v24.14.0` 以减少变量；升级到其他 Node 24.x 应单独验证，不与数据切换同时进行。

### FFmpeg / FFprobe

审计到同一工具包中的：

- FFmpeg `N-118457-gd6a364e7c3-20250210`
- FFprobe `N-118457-gd6a364e7c3-20250210`

恢复时应将：

- `FFMPEG_PATH=<ffmpeg-bin>/ffmpeg.exe`
- `FFPROBE_PATH=<ffmpeg-bin>/ffprobe.exe`

显式注入启动环境，并在启动网站前分别执行 `-version` 检查。不要在接管阶段顺便升级 FFmpeg。

## 5. Required V1 configuration

推荐的首次接管变量：

```text
PORT=48101
HOST=0.0.0.0
PHOTOS_DIR=<media-root>
DATA_DIR=<v1-runtime-root>
THUMBNAILS_DIR=<v1-runtime-root>/video-thumbnails
HLS_DIR=<v1-runtime-root>/hls
TRASH_DIR=<recycle-root>
FFMPEG_PATH=<ffmpeg-bin>/ffmpeg.exe
FFPROBE_PATH=<ffmpeg-bin>/ffprobe.exe
DUPLICATE_BATCH_SIZE=100
ALLOW_REMOTE_DELETE=0
```

推荐使用项目外的受控 PowerShell 启动环境、任务计划程序或服务管理器注入这些值，再执行 `<project-root>/server.js`。不要把真实路径写回 Git 跟踪的 `.env.example` 或启动脚本。

由于当前 `start-server-48101.cmd` 会覆盖 `DATA_DIR`，在未单独批准参数化脚本前，不应使用它来启动外部 `<v1-runtime-root>`。可以使用预设环境变量后的 `node server.js`，或不覆盖路径的 `start-site.*`。

## 6. Data migration strategy

### Recommended strategy: independent V1 runtime copy

推荐把旧运行数据复制到 Git 外的 `<v1-runtime-root>`，让 V1 只写新副本：

1. 保持 `<media-root>` 原地不动，V1 与旧项目使用同一只读浏览来源。
2. 停止旧服务和所有扫描/查重 worker。
3. 再次确认端口、Node 进程以及 `gallery.db-wal`/`gallery.db-shm` 状态。
4. 对旧 `gallery.db` 建立只读备份或停机文件副本。
5. 将必需和选定的条件数据复制到空的 `<v1-runtime-root>`。
6. 对源/目标文件记录大小、时间和 SHA-256；不删除源文件。
7. 只对目标副本执行数据库完整性和应用兼容性验证。
8. 使用 `ALLOW_REMOTE_DELETE=0` 启动 V1，再执行只读页面/API 验证。
9. 完成只读验证后，才逐项验证收藏、最近、扫描和查重等写入行为。

这套方案需要约 1.1–2.1 GiB 额外空间，取决于是否复制缓存。审计时新代码所在磁盘约有 821 GiB 可用，容量充足，但实施前仍需重新检查。

### Copy classification

本次必须做：

- 复制 `gallery.db`，保留收藏、最近、扫描状态和媒体哈希。
- 记录源/目标哈希和复制时间。
- 保持 `<legacy-data-root>` 完整不变。

建议现在做：

- 复制 `video-metadata.json`，减少 FFprobe 重复读取。
- 当 `PHOTOS_DIR` 保持同一 `<media-root>` 时复制 `thumbnails/`，减少重新生成和原图读取。
- 复制 `video-thumbnails/` 作为现有 poster 文件缓存，但不能把它视为 poster 映射缺陷的修复。

可以不复制并让系统重建：

- `highlight-carousel/`
- `highlight-carousel.json`
- 空 `hls/`
- `logs/`

默认不迁移：

- `gallery.json`。当前 JSON 恢复函数明确禁用，旧 API 返回 410；如需审计保留，应放在独立归档而不是 V1 runtime 主路径。

### Alternative: direct mount of legacy DATA_DIR

可以把 V1 `DATA_DIR` 直接指向 `<legacy-data-root>`，但不推荐作为长期方案，因为：

- V1 启动会打开并写入旧数据库；
- 数据层会启用 WAL 并执行 schema/index 保证语句；
- 日志、缩略图和轮播缓存会继续写入旧项目目录；
- 旧项目不再是严格只读备份；
- 回滚时无法区分新旧项目产生的数据变化。

该方案只有在用户明确接受旧 data 被 V1 写入、并完成独立备份后才可考虑。

## 7. Media and generated-file considerations

### PHOTOS_DIR

- 优先保持 `<media-root>` 的物理路径不变，避免全量重新索引和缓存失配。
- 不复制数十万原始媒体到项目目录。
- 切换前确认 V1 运行账户对媒体有读取权限，对真实媒体删除默认关闭。
- 扫描、查重和缩略图生成会产生显著磁盘 I/O；首次只读验证时不要触发这些任务。

### Image thumbnails

- 位置固定为 `DATA_DIR/thumbnails/<width>/<hash>.jpg`。
- 旧缓存约 607 MiB，复制可避免大量重新生成。
- 缓存键包含源文件路径/大小/mtime 语义；若 `<media-root>` 改变，旧缓存可能无法复用。
- 即使复制，也应保留缺失时按需生成的能力，并建立容量统计/清理策略。

### Video posters

- 文件位置默认为 `DATA_DIR/video-thumbnails/<hash>.jpg`。
- 旧缓存约 39 MiB。
- 当前已知问题：新进程可能没有从 SQLite 恢复 poster 源路径映射，导致 poster URL 返回 404。
- 复制 poster 文件不能保证绕过该映射检查；V1.4 不修复此问题，只在验收中记录。

### HLS

- 默认位置 `DATA_DIR/hls`，可由 `HLS_DIR` 覆盖。
- 旧 HLS 目录当前为空，无需迁移。
- HLS 会持续占用磁盘，只能按需手工生成，并应设置容量和清理规则。

## 8. Risks

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 复制运行中的 SQLite | 数据不一致或损坏 | 切换前停旧服务/worker，确认无 WAL/SHM，再复制 |
| V1 直接使用旧 DATA_DIR | 旧备份被写入，回滚边界消失 | 使用独立 `<v1-runtime-root>` |
| Node 不在 PATH | 新脚本无法启动 | 启动前验证 Node 24.x 和 `node:sqlite` |
| 同端口并行启动 | `EADDRINUSE` | 新旧服务不得同时监听 48101 |
| `ALLOW_REMOTE_DELETE=1` | 远程用户可移动真实媒体 | 首次接管固定为 0，并保留可信网络限制 |
| 媒体根路径改变 | 索引/缩略图缓存失配，大量重扫 | 首次接管保持原 `<media-root>` |
| 缩略图全量重建 | CPU、磁盘 I/O、耗时和容量增加 | 条件复制旧缓存，后续按需生成 |
| poster 映射缺陷 | 视频封面 404 | 记录为已知问题，不在 V1.4 审计阶段修复 |
| `gallery.json` 被误当主数据 | 误恢复旧索引逻辑 | 不放入 V1 runtime 主路径 |
| 跨盘回收站移动 | `fs.renameSync` 失败 | `TRASH_DIR` 与媒体尽量同盘，先用测试文件验证 |
| 日志/缓存无限增长 | 磁盘逐渐占满 | 后续增加容量统计、告警和清理策略 |
| 扫描/查重误触发 | 大量 CPU 和磁盘读取 | 首次验证只读，重任务需单独确认且保持单并发 |

## 9. Proposed cutover sequence

以下是后续实施步骤，本阶段没有执行：

1. 记录旧服务最后运行状态和旧数据清单。
2. 停止旧服务、扫描和查重任务。
3. 确认端口 48101 空闲，且数据库没有 WAL/SHM 活动文件。
4. 创建空的 `<v1-runtime-root>`，确认剩余磁盘空间。
5. 复制必需数据和选择的缓存，不覆盖任何已有目标文件。
6. 校验源/目标大小和 SHA-256。
7. 配置 Node、FFmpeg、`PHOTOS_DIR`、`DATA_DIR`、`TRASH_DIR` 和安全开关。
8. 在不触发扫描/查重/删除的情况下启动 V1。
9. 验证首页、配置、统计、目录、搜索、图片、视频 Range、收藏读取和日志写入。
10. 记录 poster 已知问题和任何路径差异。
11. 只在只读验证通过后，使用隔离测试数据验证写入型功能。
12. 确认 V1 稳定后再决定是否配置开机自启；旧项目和旧 data 暂不删除。

## 10. Rollback plan

推荐复制策略下的回滚步骤：

1. 停止 V1 服务和所有 V1 worker。
2. 确认没有真实媒体移动任务正在执行。
3. 保留失败现场的 `<v1-runtime-root>`，不要覆盖或删除，供诊断。
4. 恢复旧项目原启动方式、旧 Node/FFmpeg 和 `<legacy-data-root>`。
5. 启动旧服务前确认端口 48101 空闲。
6. 验证旧首页、SQLite 统计、目录、搜索、图片和视频。
7. 如果 V1 曾获准执行媒体删除/移动，必须先核对 `<recycle-root>` 和数据库记录；单纯切回旧数据库不会自动恢复物理文件。

因为推荐方案不修改旧项目或旧 data，代码/数据库回滚只需切换启动入口。共享的 `<media-root>` 仍是最大风险，因此 V1 首次接管必须保持 `ALLOW_REMOTE_DELETE=0`，并禁止删除类验证。

## 11. Decisions required before implementation

进入 V1.4 实施前需要用户确认：

1. 是否采用推荐的独立 `<v1-runtime-root>`，而不是直接挂载旧 data。
2. 是否复制约 607 MiB 图片缩略图和约 39 MiB 视频 poster 缓存。
3. 是否复制 `video-metadata.json`。
4. `TRASH_DIR` 的目标位置以及是否与媒体同盘。
5. 正式 Node 24.x 的安装/托管方式。
6. 首次接管是否继续保持远程删除完全关闭（推荐：是）。
7. 后续是否需要任务计划程序或服务管理器托管；这应在手工切换稳定后单独进行。

在这些选择确认前，不应复制生产数据或启动 V1。
