# V1.4.5 Runtime Final Check

## Final cache strategy

| 类型 | 当前策略 | Runtime目录 |
|---|---|---|
| 图片缩略图 | 短期禁用主动生成；兼容URL缺失时返回原图 | `D:\GalleryRuntime\thumbnails` |
| 视频poster | 用户访问时按需生成并复用 | `D:\GalleryRuntime\video-posters` |
| HLS | 保留按需设计；当前仍使用稳定的原视频Range播放 | `D:\GalleryRuntime\hls` |

旧项目缓存不是任何一种缓存的来源。本阶段没有删除V1.4.3/V1.4.4留下的少量测试缓存。

## Image policy

Runtime配置：

```text
ENABLE_IMAGE_THUMBNAIL_GENERATION=0
```

服务端行为：

1. 如果请求的Runtime缩略图已经存在，继续兼容提供该文件。
2. 如果不存在，不调用FFmpeg生成，直接提供对应原图。
3. 灯箱和完整查看继续使用`/photos/...`原图。
4. 缩略图目录和数据库中的thumbnail URL保持不变，不修改schema或历史索引。

实际验证选择了一个未缓存图片：原图URL和thumbnail兼容URL均返回63,835 bytes，SHA256完全一致；请求前后Runtime缩略图均为40个，证明没有生成新文件。

`scripts/rebuild-gallery-cache.ps1`的图片默认上限已改为0。未来即使显式传入大于0的数量，也必须同时使用`-EnableImageThumbnailRequests`，并且Runtime配置需另行改为允许生成。

## HLS policy

### Current safe behavior

当前浏览器播放器继续通过原视频HTTP Range播放。项目没有在启动时或列表加载时生成HLS，V1.4.5也没有生成任何HLS文件。

`make-hls.ps1`只能写入配置的`HLS_DIR`或显式`-OutputRoot`，不会回退到代码仓库。

### On-demand design

“用户播放时不存在则生成”应在后续独立阶段实现为异步任务，不能在视频HTTP请求中同步运行FFmpeg：

1. 播放请求先检查确定性的HLS缓存键和`index.m3u8`。
2. 已存在则更新访问manifest并返回HLS状态/地址。
3. 不存在则建立单视频、单并发后台任务，立即让客户端继续使用原视频Range或显示处理中状态。
4. 生成到同Runtime下的临时目录，成功后原子提升到最终HLS目录。
5. 失败时清理临时文件并记录错误；不得影响原视频播放。
6. 前端只有在浏览器/播放器支持HLS时才切换到HLS，否则保持原视频Range。

这需要新的后台任务状态和播放器兼容评审，本阶段只形成设计，不修改现有API或UI。

## HLS lifecycle design

Runtime配置已加入：

```text
HLS_CACHE_EXPIRE_DAYS=7
```

本阶段该值只作为策略记录，不会触发删除。

### Access time requirement

需要显式访问记录。不能只依赖NTFS `LastAccessTime`，因为Windows可能禁用或延迟更新文件访问时间，而且播放playlist和segment时更新不一致。

推荐在Runtime维护独立manifest，例如：

```text
D:\GalleryRuntime\logs\hls-cache-access.json
```

每个HLS缓存键至少记录：`lastAccessAt`、`createdAt`、总字节数、状态和源媒体标识。它是运行元数据，不进入Git，不修改SQLite schema。

### Scheduled cleanup

需要独立定时任务，建议每日低峰运行一次，而不是随网站启动执行。清理分两阶段：

1. 默认dry-run：列出超过7天未访问的候选、文件数和总大小，只写日志。
2. 用户确认策略稳定后才启用删除模式；删除前再次验证路径必须位于Runtime HLS根目录，不跟随目录外联接。

清理任务必须具备锁、暂停标记、最大删除数量/字节上限和追加日志。V1.4.5没有创建或执行删除任务。
