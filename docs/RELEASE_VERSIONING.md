# 正式版本标识与网页更新记录

## 1. 适用范围

本规范自 v102 起适用于每次正式网页发布。历史 v99、v100、v101 只记录已确认的发布日期，不补造未记录的时分。

## 2. 正式版本号

正式版本统一使用 `v<递增版本号>-YYYYMMDD-HHmm`。时间取正式发布完成时的中国标准时间（IANA `Asia/Shanghai`，Windows `China Standard Time`，UTC+8），使用 24 小时制并补足年月日时分。

发布时间不使用开发开始时间、提交时间或隔离测试时间。同一天多次发布仍需递增主版本号并使用实际完成时分，因此完整版本号必须唯一。

## 3. 单一网页数据源

用户可见的简短更新记录由根目录 `release-notes.json` 单独维护。数组必须按发布时间倒序；第一项 `version` 必须与 `app.js` 的 `APP_VERSION` 一致。每个版本仅保留 1 至 3 条用户可理解的说明，每条不超过 30 个中文字符，不记录 Git、PID、数据库路径、服务器盘符或测试过程。

旧版本若只能确认日期，使用 `releasedAt: YYYY-MM-DD` 和 `timePrecision: date`。不得为了格式统一伪造时分。

## 4. 发布步骤

1. 递增主版本号。
2. 完成隔离检查和发布前验证。
3. 在正式发布完成所在分钟取得 `Asia/Shanghai` 时间。
4. 同步更新 `APP_VERSION`、`index.html` 中所有静态资源查询参数和 `release-notes.json` 第一项。
5. 在 `release-notes.json` 增加 1 至 3 条简短说明，并更新详细 `CHANGELOG.md`。
6. 运行 `node test-release-notes.js`，验证当前版本、缓存参数、时间顺序、文案长度和公开信息边界。
7. 普通提交、普通 push，并验证正式页面、静态资源 URL 与版本记录第一项完全一致。

不得只修改页脚版本而遗漏缓存参数或网页更新记录。正式发布时间若跨分钟，应使用新的普通提交修正，不使用 `commit --amend`。

## 5. 网页行为

设置页 `#/__settings/release-notes` 按需读取 `release-notes.json`，首页不预取，也不写访问日志或 SQLite。默认显示最近 20 个版本，更多版本由用户点击“加载更多”后显示。读取失败时保留页面并显示可重试的友好提示。
