# V2.0.1 移动端流量与按需加载审计

审计日期：2026-07-13  
范围：`index.html`、`app.js`、`styles.css`、`server.js`、`gallery-db.js`、首页轮播、目录/媒体渲染及相关 API。  
运行环境：`D:\GalleryRuntime`，端口 48102。本文只记录只读审计结果，没有修改生产数据库、媒体、Runtime 配置或运行状态。

## 1. 结论摘要

当前 SQLite API **没有递归返回子孙目录媒体**：首页只查询根 collection；目录 API 只返回当前 collection 和直接子目录；媒体 API 只查询指定 collection，并已有 `LIMIT/OFFSET`。因此“深层目录媒体被 API 递归预取”不是当前主要根因。

已确认的主要流量根因是：

1. 图片列表虽然使用 `/image-thumbnails/...` URL，但正式 Runtime 配置 `ENABLE_IMAGE_THUMBNAIL_GENERATION=0`。缓存不存在时，服务端直接把原图作为该 URL 的响应返回，因此 URL 名称不能证明传输的是预览图。
2. 首页轮播一次渲染最多 20 个带 `src` 的 `<img>`。`loading="lazy"` 不能严格阻止横向隐藏项下载。
3. 小时轮播生成在 thumbnail 生成关闭时会把候选原图复制到 `highlight-carousel`。审计时 20 个轮播文件共 157,448,709 bytes（150.15 MiB），最大单文件约 13.0 MiB；这是明确且严重的移动端流量风险。
4. 首页所有根目录卡片一次性进入 DOM，并都带图片 `src`；根目录 API 和 DOM 没有首页 `limit`。
5. 目录首批 API 最多返回 120 个媒体；图片 DOM 每批 50 个，900px 缓冲区触发下一批。该实现限制 DOM 增长，但首批元数据和图片候选仍偏大。
6. fetch 没有 `AbortController`。切换 hash 后旧请求不能取消；旧 collection 请求完成后仍可能渲染过期页面。
7. 懒加载关闭时，目录中全部 video 会使用 `src + preload="metadata"`，可能并发产生多个视频 Range/metadata 请求。

## 2. 首页请求链路

首次脚本执行顺序如下：

1. `GET /api/recent`，服务端上限 20，前端保留 10。
2. `GET /api/favorites`，服务端及前端上限 100。
3. 上述两项结束后，并行请求：
   - `GET /api/collections/root`：返回全部根 collection，无 limit；不包含媒体数组或子孙 collection。
   - `GET /api/highlights`：通常返回 20 项；缓存失效时会同步重建轮播文件。
4. 首页 render 后，`POST /api/access-log` 记录访问。

首页本身不会调用 `/api/collections/:id` 或 `/api/media`，也未发现二级、三级目录 API 预取。首页 API 返回的 collection 字段包含 cover/coverThumb、直接媒体计数、累计媒体计数、descendantCount 和 childCount；这些是数据库轻量元数据，不需要加载子目录媒体来计算。

## 3. API 数据范围与分页

| API | 当前范围 | 上限/分页 | 风险判断 |
|---|---|---|---|
| `/api/collections/root` | 所有 `parent_id IS NULL` collection | 无 | 非递归，但根项很多时响应和 DOM 无界 |
| `/api/collections/:id` | 当前 collection + 直接 children | 无 children limit | depth 等价于 0/直接一层，不含孙级媒体；超大单层仍可能过多 |
| `/api/media` | 指定 collection 的直接媒体 | 默认 100，1–500；offset | 已分页；offset 大时效率下降，前端首批 120 偏大 |
| `/api/search` | collection 与 media 各自查询 | 默认 50，最大 200；前端 80 | 不递归，但一次最多可返回 160 条结果 |
| `/api/highlights` | 最多 20 项 | 固定 selection | 元数据小，关联图片文件异常大 |
| `/api/recent` | 用户标记 | 20 | 有界 |
| `/api/favorites` | 用户标记 | 100 | 有界；卡片可能触发最多 100 张图片 |

数据库索引 `collections(parent_id, sort_order)` 和 `media(collection_id, type, sort_order)` 支持当前浅层查询。collection 的 childCount 采用相关子查询；根项或单层 children 极多时可能形成重复计数成本，但不是 N+1 网络请求。

## 4. 图片加载行为

代码中未发现 `new Image()`、`<link rel=prefetch/preload>`、CSS `background-image/url(...)` 媒体、隐藏预取容器或服务端路由预取。

发现的图片入口：

- 首页轮播：最多 20 个真实 `src`，`loading="lazy"`。
- 首页目录、收藏、最近观看：卡片直接设置 `src`，使用 `loading="lazy"`。
- 目录图片：优先 `previewThumb || thumb || src`，直接设置 `src`；懒加载开启时 `loading="lazy"`。
- 搜索/查重卡片：优先 detailThumb/thumb，最后回退 src。
- 灯箱：仅用户点击时给 `#lightboxImage.src` 赋原图 URL，关闭时清空，符合原图按需查看目标。

`loading="lazy"` 是浏览器提示而不是严格调度。尤其轮播横向隐藏项、收藏/最近和根目录的大量卡片仍可能提前请求。当前没有逐图片 `IntersectionObserver`；已有 observer 只负责每次向 DOM 追加 50 个图片节点。

## 5. 预览与轮播缓存

图片 thumbnail URL 的键由源绝对路径、size、mtimeMs 和目标宽度的 SHA-1 构成，具备源文件变更失效能力。支持 480/720/960 三档 JPEG，FFmpeg 同步生成。

但正式 Runtime 关闭图片 thumbnail 生成。审计时：

- `D:\GalleryRuntime\data\thumbnails`：40 个文件，2,134,778 bytes（2.04 MiB）。
- `D:\GalleryRuntime\data\highlight-carousel`：20 个文件，157,448,709 bytes（150.15 MiB）。
- `highlight-carousel.json`：20 项，生成时间 `2026-07-13T07:00:04.951Z`。

轮播每小时清空并重建目录；当 960 thumbnail 不存在且生成关闭时，代码复制原图。这不会修改原媒体，但产生高磁盘 I/O、Runtime 写入和巨额网络传输。现有清理只针对轮播自身目录，未误删 thumbnail/poster/HLS，但没有容量统计 API、dry-run 或预览缓存人工维护命令。

## 6. 视频加载

默认懒加载开启时，video 使用 `data-src`、`preload="none"` 和 poster；pointerdown/play 后才设置 src 并调用 load，服务端支持 HTTP Range。没有首页自动播放，也不会自动生成 HLS。

风险：用户关闭懒加载后，所有 video 改为 `src + preload="metadata"`。页面离开时 render 会替换 DOM，但没有显式 pause、removeAttribute(src)、load 释放流程；正在进行的媒体请求能否立即停止取决于浏览器。

## 7. HTTP 缓存、压缩和条件请求

- 图片/视频：`Cache-Control: public, max-age=604800`，支持 Range。
- JS/CSS：`Cache-Control: public, max-age=3600`。
- HTML/JSON 及其他：`no-store`。
- 未设置 ETag、Last-Modified，未处理 If-None-Match/If-Modified-Since。
- 未启用 gzip/Brotli；JPEG/WebP/MP4 也没有被错误地重复压缩。
- 静态源文件大小：index.html 3,639 bytes、app.js 87,027 bytes、styles.css 21,332 bytes（文件大小，不是浏览器传输实测）。

## 8. 页面切换与内存

轮播 timer 和图片批次 observer/scroll listener 会在 render 时清理，这是正确的。灯箱关闭会清空 img src。没有 object URL。

不足之处：普通 fetch 不可取消；collection 加载以全局 `sqliteLoading` 标识但没有请求代次校验。快速切换路由时，较旧 promise 可能在新页面后完成并调用 `renderCollection`。视频节点销毁前也没有显式释放 src。

## 9. 优化前基线

### 已实测/只读统计

- 首页关联轮播文件：20 个，150.15 MiB。
- 已有 image thumbnail：40 个，2.04 MiB。
- 静态源文件合计：111,998 bytes（未含 HTTP headers，未代表实际浏览器传输）。

### 代码可确定

- 首页固定 API GET 数：4；渲染后另有 1 个 access-log POST。
- 首页深层目录 API 请求：0。
- 轮播 DOM 图片上限：20。
- 收藏上限 100、最近上限 10；实际项目项数未知。
- 目录首批媒体上限：120；初始图片 DOM 上限：50。

### 本轮无法实测

应用内浏览器安全策略拒绝访问 48102，且未绕过该限制。因此以下数据不得填写估算值：浏览器实际 HTTP 请求数、实际图片请求数、首页原图请求数、API 响应字节、总传输量、首屏时间、屏幕下方 lazy 图片实际下载数量。

人工 DevTools 补验时应：清空缓存并勾选 Disable cache；分别用移动端尺寸和网络限速重载 `#/`；导出 HAR；记录 request/transfer/resource size、DOMContentLoaded/LCP，并按 `/photos/`、`/image-thumbnails/`、`/highlight-carousel/`、`/api/` 分类。随后只点击一级目录，确认未出现二/三级 API；进入目录后滚动并验证批次行为；未点击视频前确认没有视频 Range。

## 10. 推荐实施边界

### 本次必须做

1. 首页显式 limit，并为 root 查询增加向后兼容的分页参数。
2. 列表使用真正预览文件；不再以 thumbnail URL 回退传输原图。
3. 采用单一通用预览规格，按请求生成到独立 Runtime 缓存，不做全库任务。
4. 图片 `data-src + IntersectionObserver`；轮播只加载当前项，最多预备下一项。
5. 目录首批降至 40–60，保留分批追加；修复混合图片/视频分页遗漏风险。
6. 路由切换 AbortController + 请求代次校验；视频始终 `preload=none` 并在离页时释放。
7. 缓存统计、dry-run、人工清理；清理根严格限定预览目录，不自动批量删除。
8. 保持 schema、端口、Runtime 启动体系、原媒体、ZeroTier、防火墙和 HLS 策略不变。

### 建议现在做

- ETag/Last-Modified/304；文本 gzip。
- 预览生成去重与小并发上限，避免同一图片并发触发多个 FFmpeg。
- 首页收藏/最近也采用同一严格 lazy loader。

### 以后规模变大再做

- 稳定游标分页、预览后台队列、LRU 自动淘汰/告警、虚拟列表、CDN/对象存储。

## 11. 验收重点

实现后必须用隔离 Runtime 验证：只请求可见预览、访问 20 张只生成不超过约 20 个缓存、原媒体 SHA-256 与 mtime 不变、数据库 schema 不变、视频未点击无 Range、路由切换旧响应不渲染、缓存清理 dry-run 不越界。生产 Runtime 只允许只读容量核对；任何生成或清理测试需另行授权。
