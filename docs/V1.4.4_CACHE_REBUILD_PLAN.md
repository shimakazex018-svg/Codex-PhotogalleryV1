# V1.4.4 Runtime Cache Rebuild Plan

实施日期：2026-07-12

## Goal and boundary

V1.4.4 建立只属于 `D:\GalleryRuntime` 的缓存来源，修复新进程 poster 404，并提供可暂停、可恢复、有日志的小批量生成工具。本阶段只执行20个图片缩略图请求和3个视频poster请求；没有全量重建、HLS生成、扫描、查重、媒体修改或数据库schema变更。

## Current cache paths

| 类型 | 唯一Runtime来源 | 当前实现 |
|---|---|---|
| 图片缩略图 | `D:\GalleryRuntime\thumbnails` | `DATA_DIR\thumbnails` directory junction指向该目录 |
| 视频poster | `D:\GalleryRuntime\video-posters` | `POSTER_DIR`由启动器映射到服务端`THUMBNAILS_DIR` |
| HLS | `D:\GalleryRuntime\hls` | `HLS_DIR`直接注入服务端和手工生成脚本 |

不读取或复制旧项目的缩略图、poster或HLS。

## Generation logic analysis

### Image thumbnails

- URL格式：`/image-thumbnails/{480|720|960}/<sha1>.jpg`。
- 文件键基于源路径、大小、mtime和目标宽度。
- 缓存缺失时，服务端从SQLite的`thumb/detail_thumb/carousel_thumb`回查`src`，验证路径位于`PHOTOS_DIR`后调用FFmpeg。
- 已存在文件直接复用。

### Video poster

- URL格式：`/video-posters/<sha1>.jpg`。
- 文件键基于源路径、大小和mtime。
- 原实现只读取进程内`videoPosterSources`，重启后直接请求已存poster URL会404。
- V1.4.4在内存未命中时，按`media.poster`从Runtime SQLite只读查询`src`，再限制到`PHOTOS_DIR`并生成poster。
- 不增加表、字段或索引，不使用旧缓存兼容路径。

### HLS

- 静态服务继续只读取`HLS_DIR`。
- `make-hls.ps1`不再默认写入仓库内`data\hls`；必须设置`HLS_DIR`或显式传`-OutputRoot`。
- HLS仍是单视频手工任务。本阶段没有生成HLS，也没有实现全量HLS队列。

## Controlled rebuild tools

`scripts/rebuild-gallery-cache.ps1`默认上限为20张图片和3个视频poster，HLS为0且不支持隐式生成。

运行状态：`D:\GalleryRuntime\logs\cache-rebuild-state.json`。

逐项日志：`D:\GalleryRuntime\logs\cache-rebuild.log`。

状态文件记录成功URL和失败信息。每个项目完成后通过临时文件替换保存状态，重新运行时跳过已完成URL。

请求暂停：

```powershell
.\scripts\pause-cache-rebuild.ps1 -RuntimeRoot D:\GalleryRuntime
```

任务在目录发现或每个媒体请求前检查 `D:\GalleryRuntime\logs\cache-rebuild.stop`。

恢复：

```powershell
.\scripts\rebuild-gallery-cache.ps1 -Resume
```

零媒体控制测试确认：暂停状态拒绝启动并返回非零，`-Resume`移除停止标记并正常完成。

## Full rebuild design — not executed

当前小批量脚本不能直接视为47万图片的全量调度器。进入全量前必须单独实现并评审：

1. 从SQLite只读导出稳定、分页的缓存清单，避免反复遍历全部collection。
2. 图片按宽度和目录分批，例如每批100–500，默认单并发。
3. poster按20–50个一批，FFmpeg保持单并发并设置单项超时。
4. 每批前检查Runtime磁盘剩余空间，低于门限立即暂停。
5. 状态保存清单版本、游标、成功、失败、重试次数和最后错误。
6. 失败项目限制重试次数，不能无限循环。
7. 每批完成后记录耗时、文件数和磁盘增长，并设置人工观察窗口。
8. HLS不纳入默认全量重建；只对明确选定视频逐个生成，并单独设置容量预算和清理规则。
9. 全量任务只能在用户再次授权后运行，不得复用本阶段授权。

## Rollback

代码可回滚到V1.4.3提交。Runtime缓存是可重建数据，但本阶段不自动删除已生成文件；如需清理，必须先列清单并获得单独授权。数据库与媒体无需回滚，因为本阶段没有修改它们。
