# V1.4.1 Runtime Implementation Plan

## 1. Confirmed design decisions

本计划固化以下已确认要求：

1. V1 必须完全独立运行，未来删除旧项目后仍可启动、查询和维护。
2. V1 不得读取或写入旧项目目录。
3. 原始媒体保持当前物理位置，不复制到代码或 runtime。
4. 图片缩略图不迁移，在新 runtime 中重新生成。
5. 视频 poster 不迁移，在新 runtime 中重新生成。
6. HLS 不迁移，在新 runtime 中按需重新生成。
7. 新监听端口为 `48102`。
8. 代码与数据彻底分离。
9. 所有端口、媒体、数据、工具和回收路径均由外部配置提供，不写死在代码仓库。

本文件只描述实施方案。本阶段没有创建真实 runtime、复制数据、修改脚本、打开数据库或启动网站。

## 2. Path placeholders and local mapping

为避免把真实机器路径提交到 Git，仓库文档和配置模板使用：

| 占位符 | 本机实施含义 |
|---|---|
| `<project-root>` | 当前 V1 代码目录 |
| `<runtime-root>` | 用户确认的独立 GalleryRuntime 目录 |
| `<media-root>` | 用户确认的现有原始照片/视频目录 |
| `<legacy-data-root>` | 旧项目 data，仅用于一次性迁移源 |
| `<recycle-root>` | 与媒体同盘的受控回收目录 |
| `<node-exe>` | Node 24.x 可执行文件 |
| `<ffmpeg-exe>` | FFmpeg 可执行文件 |
| `<ffprobe-exe>` | FFprobe 可执行文件 |

真实路径只写入 `<runtime-root>/config/gallery.env`。该文件位于代码仓库外，不进入 Git。

## 3. Runtime directory design

目标结构：

```text
<runtime-root>/
├─ config/
│  └─ gallery.env                 # 实际运行配置，不进入 Git
├─ data/                          # DATA_DIR
│  ├─ gallery.db                  # 从旧环境迁移的主数据库
│  ├─ video-metadata.json         # 可选迁移的 FFprobe 缓存
│  ├─ thumbnails/                 # 新环境重新生成
│  │  ├─ 480/
│  │  ├─ 720/
│  │  └─ 960/
│  ├─ video-thumbnails/           # 视频 poster，新环境重新生成
│  ├─ highlight-carousel/         # 新环境重新生成
│  ├─ highlight-carousel.json     # 新环境重新生成
│  ├─ hls/                        # 按需重新生成
│  └─ logs/                       # 新环境重新记录
├─ backups/
│  └─ database/                   # V1 数据库版本化备份
└─ staging/
   └─ database/                   # 复制、校验、提升前的临时位置
```

设计说明：

- `data/` 必须位于 Git 仓库外。
- `gallery.db` 是唯一必须迁移的数据文件。
- `staging/` 与 `data/` 建议位于同一磁盘，校验完成后可原子重命名/移动。
- `backups/` 不参与应用运行，禁止与当前 `gallery.db` 混用。
- `logs/`、缩略图、poster、轮播和 HLS 的实际位置由当前服务从 `DATA_DIR` 派生，目录设计必须尊重这一事实。
- runtime 根目录只授予运行账户和管理员所需权限；普通网络访问者不应获得文件系统权限。

## 4. Actual environment-file design

实际配置文件：

```text
<runtime-root>/config/gallery.env
```

建议内容格式：

```dotenv
# Listener
PORT=48102
HOST=0.0.0.0

# Code-independent runtime paths
PHOTOS_DIR=<media-root>
DATA_DIR=<runtime-root>\data
THUMBNAILS_DIR=<runtime-root>\data\video-thumbnails
HLS_DIR=<runtime-root>\data\hls
TRASH_DIR=<recycle-root>

# Executables
NODE_EXE=<node-exe>
FFMPEG_PATH=<ffmpeg-exe>
FFPROBE_PATH=<ffprobe-exe>

# Resource and safety controls
DUPLICATE_BATCH_SIZE=100
ALLOW_REMOTE_DELETE=0
```

### File grammar

V1.4.1 启动器应支持：

- UTF-8 文本；
- 空行；
- 以 `#` 开头的整行注释；
- 每行第一个 `=` 左侧为变量名、右侧为完整值；
- 路径含空格和中文，不要求额外引号；
- 拒绝重复变量、空变量名和未知危险键；
- 不执行文件中的 PowerShell、CMD、变量替换或命令替换。

禁止把 `.env` 当作脚本 `Invoke-Expression`，避免配置注入。

### Required and optional keys

| 变量 | 启动器要求 | 说明 |
|---|---|---|
| `PORT` | 必需 | V1.4 首次值为 `48102`；必须为 1–65535 整数 |
| `HOST` | 必需 | 当前继承值 `0.0.0.0` |
| `PHOTOS_DIR` | 必需 | 必须存在、可读、位于项目外 |
| `DATA_DIR` | 必需 | 必须位于项目外，运行账户可写 |
| `THUMBNAILS_DIR` | 必需 | 当前代码中表示视频 poster 目录 |
| `HLS_DIR` | 必需 | 必须位于 runtime 中 |
| `TRASH_DIR` | 必需 | 推荐与媒体同盘；首次删除功能仍禁用 |
| `NODE_EXE` | 启动器必需 | Node 24.x；服务端本身不读取此变量 |
| `FFMPEG_PATH` | 必需 | FFmpeg 可执行文件 |
| `FFPROBE_PATH` | 必需 | FFprobe 可执行文件 |
| `DUPLICATE_BATCH_SIZE` | 必需 | 初始保持 `100` |
| `ALLOW_REMOTE_DELETE` | 必需 | 首次恢复必须为 `0` |

`NODE_EXE` 是启动器变量，启动 Node 前不需要传给服务端。其他变量以当前进程环境传递给 `server.js` 和查重 worker。

### Security and storage

- `gallery.env` 不存密码或 Token，但仍包含真实磁盘布局，应限制 ACL。
- 不把 `gallery.env` 复制到项目根目录。
- 不在日志中打印完整真实路径；预检报告可以显示“存在/可读/可写”状态和脱敏后的尾部路径。
- `ALLOW_REMOTE_DELETE=0` 必须由启动器强制校验；首次恢复阶段不允许配置为 `1` 或 `true`。

## 5. Startup-script modification design

### Canonical PowerShell launcher

下一实施阶段建议新增：

```text
<project-root>/start-gallery.ps1
```

参数设计：

```powershell
start-gallery.ps1 -EnvFile <runtime-env-file>
start-gallery.ps1 -EnvFile <runtime-env-file> -PreflightOnly
```

也可允许 `GALLERY_ENV_FILE` 指定配置文件；如果参数和环境变量都缺失，启动器必须失败，不能退回项目内 `data`/`photos`。

启动器职责：

1. 解析但不执行 `gallery.env`。
2. 验证所有必需键和禁止键。
3. 规范化路径并拒绝 `DATA_DIR` 位于 `<project-root>` 或 `<legacy-project-root>`。
4. 验证 `PHOTOS_DIR` 存在且可读。
5. 验证 `DATA_DIR` 存在、可写并包含预期 `gallery.db`。
6. 验证 `PORT=48102` 且端口未被占用。
7. 验证 `NODE_EXE`、`FFMPEG_PATH`、`FFPROBE_PATH` 存在。
8. 验证 Node 主版本和 `node:sqlite`。
9. 验证 runtime 磁盘剩余空间。
10. 强制首次恢复阶段 `ALLOW_REMOTE_DELETE=0`。
11. 以进程级环境变量启动 `NODE_EXE <project-root>/server.js`。
12. 将工作目录设为 `<project-root>`，但不在项目内写运行数据。

`-PreflightOnly` 必须在完成检查后退出，不打开数据库、不创建服务监听、不生成缓存。

### CMD wrapper

可新增薄包装：

```text
<project-root>/start-gallery.cmd
```

职责仅限：

- 接收第一个参数作为 env 文件路径，或读取 `GALLERY_ENV_FILE`；
- 调用 PowerShell `start-gallery.ps1 -EnvFile ...`；
- 返回 PowerShell 的退出码。

CMD 不解析 `.env`，不写死端口和路径。

### Network/firewall script

现有 `fix-network-access-48101.*` 与旧端口绑定。后续应新增或改造为参数化网络脚本：

```text
configure-gallery-network.ps1 -EnvFile <runtime-env-file>
```

它应从配置中读取 `PORT`，显示将修改的防火墙规则并要求管理员明确执行。网络脚本与普通启动脚本分离，启动网站不得自动修改防火墙。

### Existing launchers

- `start-server-48101.cmd` 在 V1.4 完成前保留，用于代码历史和紧急诊断，但不得作为新 runtime 入口。
- `start-site.cmd`、`start-site.ps1` 不加载 `.env`，也不能作为完全配置化生产入口。
- 新启动器稳定并形成回滚标签后，是否归档旧启动脚本应另行审计，不在 V1.4.1 设计阶段删除。

## 6. Data migration decisions

### Files to migrate

必须迁移：

- `<legacy-data-root>/gallery.db` → `<runtime-root>/data/gallery.db`

建议迁移：

- `<legacy-data-root>/video-metadata.json` → `<runtime-root>/data/video-metadata.json`

理由：媒体物理路径保持不变，已有 FFprobe 缓存可减少对约 2,000 个视频的重复探测。它属于可重建缓存；如果复制前校验失败，可以放弃迁移并让新环境逐步重建。

明确不迁移：

- `gallery.json`
- `thumbnails/`
- `video-thumbnails/`
- `highlight-carousel/`
- `highlight-carousel.json`
- `hls/`
- `logs/`

### Why thumbnails and HLS are rebuilt

- 新 runtime 应形成自己的缓存生命周期，与旧项目完全解耦。
- 缩略图缓存约数百 MiB，复制会携带历史无效项且无法解决淘汰问题。
- 视频 poster 文件即使复制，也无法修复已知的 poster 源路径映射缺陷。
- 旧 HLS 当前为空；后续只对明确需要的视频按需生成。

### Database staging and promotion

实施时建议：

1. 确认旧服务和 worker 全部停止。
2. 确认旧库旁没有活动 `-wal`/`-shm`。
3. 先将数据库复制到 `<runtime-root>/staging/database/`。
4. 记录源文件大小、mtime 和 SHA-256。
5. 记录目标文件大小和 SHA-256，必须完全一致。
6. 在 staging 副本上执行只读完整性检查。
7. 将校验通过的副本提升到 `<runtime-root>/data/gallery.db`。
8. 源数据库保持不变，直到 V1 稳定并完成独立备份。

禁止直接把旧 `DATA_DIR` 配置给 V1。

## 7. Runtime creation sequence

以下步骤供下一实施阶段执行，本阶段没有执行：

### Phase A: directory and permissions

1. 创建 `<runtime-root>`。
2. 创建 `config`、`data`、`backups/database`、`staging/database`。
3. 验证 runtime 不位于 Git 仓库或旧项目目录内。
4. 设置运行账户 ACL。
5. 检查 runtime 磁盘剩余空间和媒体盘可用空间。

### Phase B: configuration

1. 从跟踪的模板生成 runtime 外部 `gallery.env`。
2. 写入已确认的端口、媒体、data、工具和回收路径。
3. 设定 `ALLOW_REMOTE_DELETE=0`。
4. 使用预检模式验证，不启动网站。

### Phase C: data copy

1. 再次确认旧服务停止。
2. 备份旧数据库。
3. 复制并校验 `gallery.db`。
4. 可选复制并校验 `video-metadata.json`。
5. 保持所有缓存目录为空。

### Phase D: first start

1. 确认旧端口 `48101` 不再作为 V1 入口。
2. 确认 `48102` 空闲。
3. 使用新参数化启动器启动 V1。
4. 首次只验证读取，不触发扫描、查重、删除、HLS 或全量缩略图生成。
5. 验证首页、SQLite 统计、目录、搜索、图片原图和视频 Range。
6. 记录缩略图按需生成和 poster 已知问题。

### Phase E: controlled cache regeneration

只读功能稳定后：

1. 先抽样生成少量图片缩略图，监测 CPU、磁盘和耗时。
2. 视频 poster 单独抽样并记录 404 映射问题。
3. 不自动全量生成 HLS；只对测试视频按需生成。
4. 不一次性访问全部媒体，避免浏览器 DOM、网络和磁盘 I/O 激增。

## 8. Acceptance criteria

V1.4 运行恢复完成必须同时满足：

- V1 不读取或写入 `<legacy-project-root>`。
- 旧项目目录暂时离线/重命名后，V1 仍能启动和查询。
- `PHOTOS_DIR` 指向现有 `<media-root>`。
- `DATA_DIR` 指向 `<runtime-root>/data`。
- 当前端口是配置中的 `48102`，没有代码硬编码依赖。
- Node、FFmpeg、FFprobe 均从配置验证通过。
- 数据库源/目标哈希记录完整。
- SQLite 统计、目录、搜索、收藏/最近读取一致。
- 新缩略图、日志、轮播和 HLS 只写入 `<runtime-root>`。
- Git 工作区在运行后保持干净。
- `ALLOW_REMOTE_DELETE=0`。
- 旧 data 和旧项目在用户确认前未删除。

## 9. Rollback design

在旧项目删除前：

1. 停止 V1 和全部 worker。
2. 保留 `<runtime-root>` 现场，不删除失败数据。
3. 确认媒体没有被移动或删除。
4. 使用旧项目、旧 data 和旧端口恢复旧服务。
5. 验证旧首页、数据库统计、目录、搜索、图片和视频。

在未来删除旧项目前必须额外完成：

- 至少一份 `<runtime-root>/data/gallery.db` 的离线备份；
- V1 连续稳定运行观察期；
- 新启动器和配置的恢复演练；
- runtime 目录完整备份；
- 明确的 Git 代码版本/tag；
- 用户单独确认删除旧项目。

删除旧项目不是 V1.4 自动步骤。

## 10. Risks and controls

| 风险 | 控制 |
|---|---|
| `.env` 文件存在但未加载 | 使用参数化 PowerShell 启动器显式解析和校验 |
| 配置注入 | 不使用 `Invoke-Expression`，只解析受限 KEY=VALUE |
| Node PATH 不稳定 | `NODE_EXE` 显式配置并验证 Node 24.x/`node:sqlite` |
| `DATA_DIR` 错指项目或旧目录 | 启动器规范化路径并拒绝目录重叠 |
| 新旧服务端口冲突 | V1 使用 48102，预检端口占用 |
| 缩略图重建压垮磁盘/CPU | 按需、抽样、分阶段重建并记录容量 |
| HLS 无限增长 | 不全量生成，后续增加配额和清理策略 |
| poster 404 | 作为已知问题记录，不用迁移缓存掩盖 |
| 远程删除真实媒体 | 首次强制 `ALLOW_REMOTE_DELETE=0` |
| 跨盘回收失败 | `TRASH_DIR` 与媒体同盘并用隔离文件验证 |
| 复制数据库不一致 | 停服务、检查 WAL/SHM、staging 哈希校验 |
| 旧项目过早删除 | 稳定期、备份、回滚演练和用户二次确认 |
